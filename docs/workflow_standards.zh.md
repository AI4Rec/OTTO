# OTTO 复盘项目流程规范

目标：

用一个已结榜推荐比赛，练习工业级推荐系统项目的规范流程。每个阶段都要留下明确产物：输入、处理、输出、指标、结论和下一步，而不是只留下临时 notebook 或零散命令。

## 阶段与产物

| 阶段 | 目标 | 必交付产物 | 通过标准 |
| :-- | :-- | :-- | :-- |
| 数据接入 | 确认原始数据可信、可定位、可复现读取。 | 数据集卡、registry 记录、原始文件校验记录。 | 知道数据来源、路径、大小、字段、读取方式和风险。 |
| EDA | 建立对数据和任务的第一性理解。 | EDA 报告、关键统计表、初步 insight、后续假设。 | 能讲清字段、接口、分布、异常、任务难点和 baseline 方向。 |
| 验证体系 | 建立可迭代的本地评价闭环。 | split 说明、labels 说明、metric 实现说明、metric sanity check。 | 任意实验都能得到 clicks/carts/orders/weighted recall。 |
| Baseline | 跑通最小推荐链路。 | baseline 实验卡、输出样例、valid 分数、错误分析。 | 有一个能复现的基础分数和可对比基线。 |
| 召回提升 | 扩大候选覆盖，分析召回瓶颈。 | 候选源说明、召回覆盖率报告、消融记录。 | 知道每一路召回贡献和冗余。 |
| 排序提升 | 用特征和模型重排候选。 | 特征清单、训练集说明、模型实验卡、指标对比。 | 能解释特征/负采样/模型选择的收益和代价。 |
| 高分方案学习 | 抽象优秀方案的共性。 | top solution 阅读笔记、方法地图、可复现路线图。 | 能把高分方案拆成候选、特征、排序、融合、工程优化。 |
| 个人增量 | 做一个小而完整的改进。 | 增量实验卡、对照实验、结论、失败分析。 | 有明确假设、指标变化和可讲的技术判断。 |
| 项目讲述 | 把项目变成可表达材料。 | 项目 README、架构图、指标表、故事线、Q&A。 | 能在 3-5 分钟内讲清项目，也能展开技术细节。 |

## EDA 报告最低要求

EDA 不是画几张图，而是回答“这个推荐任务到底长什么样”。报告至少包含：

- 数据来源、文件路径、读取方式。
- 字段字典：字段名、类型、含义、样例、风险。
- 数据接口：一条 session 如何表示，event 如何排序，下游代码希望读到什么 schema。
- 基础规模：session 数、event 数、aid 数、时间范围。
- 行为分布：click/cart/order 数量、比例、按时间变化。
- session 分布：长度、重复 aid、不同 type 组合、长尾情况。
- item 分布：热门 aid、长尾 aid、不同行为类型下的热度差异。
- 时间分布：训练/测试时间覆盖、是否适合时间切分。
- 数据质量：空值、异常 type、乱序、重复、超长 session。
- 对建模的启发：至少 3 条 insight，每条对应一个可验证假设。
- 后续动作：baseline、split、metric 或特征工程的具体任务。

## 实验记录最低要求

每个实验都要能回答：

- 这次改了什么？
- 为什么认为它可能有效？
- 输入数据版本是什么？
- 运行命令是什么？
- 输出在哪里？
- 指标相比基线如何？
- 成本如何：时间、内存、磁盘。
- 结论是什么：晋级、拒绝、保留观察。
- 下一个动作是什么？

## 命名建议

```text
D000_*  数据接入与数据版本
E000_*  环境或下载 smoke test
A000_*  EDA / analysis
V000_*  validation split / metric
B000_*  baseline
C000_*  candidates / recall
F000_*  features
R000_*  ranker
S000_*  submission
I000_*  insight / project narrative material
```

## 4 天冲刺中的流程要求

| 天数 | 必须留下的规范产物 |
| :-- | :-- |
| Day 1 | 数据集卡、EDA 报告、baseline 实验卡。 |
| Day 2 | validation/metric 说明、baseline 提升实验卡、指标对照表。 |
| Day 3 | top solution 阅读笔记、方法地图、个人增量候选清单。 |
| Day 4 | 个人增量实验卡、前后对比、项目讲述草稿。 |
