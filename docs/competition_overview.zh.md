# OTTO Recommender System 竞赛全局回顾

OTTO 这场比赛放在 Kaggle 推荐系统竞赛史里看，不是那种“给你一张用户物品表，训练一个模型”的比赛，而更像一次大规模工业推荐系统的压力测试：数据巨大、目标多、强时序、强共现、线上指标明确。最后胜出的不是单个漂亮模型，而是一整套候选生成、特征工程、排序、验证、资源管理的系统工程。

## 一、比赛本质

OTTO Recommender System 的任务是：给定一个被截断的用户会话，预测截断点之后用户会点击什么、加购什么、下单什么。

它有三个目标：

- `clicks`：下一次点击的商品
- `carts`：后续加入购物车的商品
- `orders`：后续下单的商品

最终分数是加权 Recall@20：

```text
0.10 * clicks + 0.30 * carts + 0.60 * orders
```

这意味着比赛表面上是三目标推荐，实际重心明显偏向 `orders`。一个方案如果 click 做得漂亮但 order 做不好，整体很难赢。

## 二、赛前准备视角

真正进入比赛前，选手需要先理解几个事实。

第一，数据非常大。`train.jsonl` 约 11GB，事件数超过 2 亿，商品数约 180 万。普通 pandas 工作流很快会吃不消，所以很多人一开始就要决定技术栈：Polars、cuDF、Numba、Spark、DuckDB、LightGBM、XGBoost、CatBoost，甚至自己写高性能共访矩阵。

第二，测试集是“截断会话”。也就是说模型不是预测一个静态用户偏好，而是要根据用户当前 session 已经发生的行为，推断下一步会发生什么。

第三，比赛没有用户画像、商品文本、价格、类目等元数据，只有匿名 `aid` 和行为序列。所以它天然逼迫大家回到推荐系统的核心：共现、序列、近期性、行为类型转换。

## 三、初期：baseline 和规则方法占主导

比赛初期，大家很快发现一个简单但强大的事实：最近看过、加购过、下单过的商品非常重要。

于是早期 baseline 大多围绕几类规则：

- 会话内最近出现的商品
- 商品全局热度
- 点击、加购、下单之间的行为权重
- item-item 共现，也就是 co-visitation matrix
- 不同时间窗口下的共现关系

这类规则方法很快就能打到相当不错的分数。后来甚至有高分帖强调“只用规则也能做到 LB 0.590”。这说明这场比赛的第一道门槛不是深度学习，而是能不能把共访矩阵和时序规则做扎实。

## 四、中期：候选生成成为主战场

进入中期后，大家逐渐形成共识：这是一个典型的两阶段推荐系统问题。

第一阶段是 candidate generation，先从 180 万商品中为每个 session 找出几十到几千个可能商品。

第二阶段是 reranking，用更复杂的特征和模型把候选排到前 20。

候选生成成了最关键的地基。强队通常会组合多种候选来源：

- session 中已经访问过的 `aid`
- 点击到点击、点击到加购、加购到下单、下单到下单等不同共访矩阵
- 不同时间窗口的共访矩阵
- 按行为类型加权的共访矩阵
- 最近行为加强
- 全局热度补充
- 神经网络召回，如 MLP、Transformer、GRU 类序列模型

冠军方案平均每个 session 生成约 1200 个候选。第 5 名方案大部分时间只生成 80 个候选，最后提升到 120 个，也拿到了非常强的成绩。这说明候选数量不是越多越好，关键是候选质量、召回覆盖和后续排序模型能否消化。

## 五、中后期：排序模型和特征工程拉开差距

当候选生成足够强之后，竞争焦点转向 reranker。

主流路线不是端到端深度模型，而是 GBDT 排序模型：

- LightGBM Ranker
- XGBoost
- CatBoost
- 多模型 ensemble

特征大致分三类：

- `session × aid` 特征：候选在共访矩阵中的排名、与会话内已见商品的关系、最后出现时间、行为类型等
- `aid` 特征：商品热度、点击/加购/下单比例、不同时间窗口热度
- `session` 特征：会话长度、重复率、最近两个事件时间差、行为结构

冠军方案做了约 200 个特征，再按目标筛到约 100 个。第 5 名方案约 400 个特征。第二名方案通过 item2item 特征组合能生成接近 5000 个候选特征，最后选几百个使用。

这一阶段的比赛已经很像工业推荐排序系统：召回、特征、排序、融合，每一块都要打磨。

## 六、验证策略：决定迭代速度

OTTO 的一个重要特点是本地验证和 leaderboard 可以做到高度相关。

很多选手采用 Radek 社区分享的本地验证划分方法，用时间切分模拟测试集截断。冠军也提到，本地验证和 LB 几乎完美相关。这个点非常关键。

如果本地 CV 不稳，选手只能靠提交试错，迭代会很慢，而且容易被 public LB 误导。

如果本地 CV 稳，选手就可以大胆做消融实验、负采样实验、特征筛选、候选组合。

所以这场比赛里，“建立可信本地验证”几乎和“建立强模型”同等重要。

## 七、竞争状态

这场比赛的竞争很激烈，但路线逐渐收敛。

早期大家比谁能快速吃下数据、做出强规则 baseline。

中期大家比候选生成，尤其是共访矩阵版本、时间窗口、行为权重、召回覆盖。

后期大家比排序特征、负采样、模型融合、推理速度和工程稳定性。

最终强方案有几个共同点：

- 没有迷信单一深度模型
- 都非常重视共访矩阵
- 都使用两阶段架构
- 都把 orders 权重放在核心位置
- 都做了本地验证
- 都在资源约束下优化计算效率
- 都有大量细致的特征工程和消融

这不是“一个 Transformer 解决一切”的比赛。深度模型有用，但更多是作为候选来源或补充特征，而不是单独统治全场。

## 八、争议与赛后

赛后 Kaggle 发布过合规更新，说明排行榜验证后仍有团队因合规问题被移除。这意味着最终名次经历过调整，也让社区对作弊检测、团队合规、账号行为等问题有不少讨论。

后来 OTTO 又发布了完整 session 数据集，包括测试集，使这场比赛从一次 Kaggle 竞赛变成一个长期可复用的推荐系统研究基准。

这点很重要：比赛结束不是数据结束。它后来成为很多推荐系统论文、baseline、工程实验的参考数据集。

## 九、核心启发

如果用一句话总结：

OTTO 证明了在大规模匿名电商 session 推荐中，强共现召回 + 稳定本地验证 + GBDT 重排 + 工程效率，往往比单纯追求复杂模型更重要。

从学习角度看，它很适合用来理解现代推荐系统的真实工作流：

```text
数据理解
-> 时间切分验证
-> 规则 baseline
-> 共访矩阵召回
-> 多路候选合并
-> 特征工程
-> GBDT/ranker 重排
-> 消融实验
-> 模型融合
-> 推理优化
```

它不是一场只考算法的比赛，更像一场“推荐系统全链路能力”的考试。你要懂数据、懂行为、懂验证、懂工程，还要懂什么东西值得复杂化，什么东西用简单规则就足够强。
